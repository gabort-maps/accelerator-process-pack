Executive project proposal

Standardised Accelerator Operating Model

Approval note for the programme team and partners. This page is intentionally concise and should be read together with the Accelerator Process Flow, which acts as the visual backbone of the proposed operating model.

Greenfield project framing Reusable across future cohorts Process first, software later

Sections

Purpose Why now What the project will deliver Scope Reference model Decision requested

Purpose

Approve a project to design and implement a standardised, auditable, and scalable operating model for the accelerator programme. The current working materials should be treated as exploratory starting work, not as an approved system. This project creates the approved operating system.

Core intent: preserve consistency across reviewers and actors, reduce process drift, and minimise administrative burden, especially on mentors.

Why now

Consistency risk

Scaling increases inconsistency if the programme relies on individual judgement and disconnected tools.

Comparability risk

Outputs are not yet guaranteed to be comparable across reviewers, startups, and cohorts.

Efficiency risk

Mentors risk carrying unnecessary administration unless the operating logic is clearly designed and standardised.

A formal model is needed before process drift becomes structural. The aim is not bureaucracy. The aim is repeatability, auditability, and low-friction execution.

What the project will deliver

Unified operating model

One end-to-end model with an Equity / Fast Track fork.

Controlled vocabulary

One standard terminology set to reduce interpretation drift.

Mandatory template suite

Standard workbooks, templates, and outputs for operational use.

KPI framework

Balanced reporting across quality, speed, outcomes, and mentor efficiency.

Light automation readiness

Future-ready data structures without committing to software build now.

HTML-ready reference structure

Document architecture that can later support interactive step explanations.

Scope

Start point
Startup selected into the accelerator
End point
Investor facilitation completed
Target scale
Up to 20–30 startups per cohort
In scope
Process design, Equity Track, Fast Track, supporting tools, templates, workbooks, KPI layer, and light automation requirements
Out of scope
Full software build, deep integrations, investor CRM, and external mentor marketplace

Project objective

To design and implement a reusable accelerator operating model that standardises the startup journey from programme entry to investor facilitation, preserves consistency across reviewers, mentors, programme staff, and decision-makers, and minimises administrative burden, especially on mentors.

Primary outcome
Standardised, auditable, scalable execution
Operating owner
Accelerator programme team
Main user groups
Startups, reviewers, mentors, programme managers, decision-makers
Unit of operation
Startup-level logic inside cohort-level constraints

Design principles

Standardised, not rigid

Structured process with controlled flexibility and guided overrides.

Auditability by design

Major decisions must be visible and defensible.

Low mentor administration

Mentors focus on support, not process management.

Cohort-aware logic

The model must work per startup and across the cohort.

Reusable model

Suitable for future cohorts without redesign.

HTML-ready structure

Documentation should support a future interactive reference layer.

Scope definition

In scope

  • End-to-end process from startup selected into programme to investor facilitation completed
  • Unified process design with a structured Equity / Fast Track fork
  • Supporting tools, templates, and workbooks
  • KPI and reporting structure
  • Light future automation requirements

Out of scope

  • Full software platform build
  • Deep automation or complex integrations
  • Investor CRM development
  • External mentor marketplace build

Target operating model

The target model is one unified process with an early fork, supported by common data structures, mandatory templates, and a shared reporting layer.

Shared entry layer

  • Startup selected into programme
  • Self-assessment
  • First 1:1 review
  • Common base data capture and startup profile foundation

Equity Track

  • Diagnosis of mentoring and support needs
  • Creation of startup demand profile and support package
  • Mentor supply standardisation and matching
  • Assignment, review, or gap handling
  • Mentoring delivery lifecycle

Fast Track

  • Investor-readiness pre-check
  • Evidence and DD / data-room requirements
  • Conditional remediation where gaps exist
  • Optional light support where justified
  • Investor facilitation handoff

Workstreams

WS1. Operating model design

Define the full target-state process and branch logic.

WS2. Template and workbook standardisation

Create and lock the mandatory operating templates.

WS3. Mentoring system design

Define diagnosis, matching, assignment, delivery, and review logic.

WS4. Fast Track and investor-readiness design

Define readiness logic, evidence logic, remediation, and facilitation outputs.

WS5. Reporting and KPI layer

Define how consistency and performance will be measured.

WS6. Light automation readiness

Define the minimum future requirements for scaling without building software now.

Proposed phasing

Phase 1. Detailed design

Approve the operating model, terminology, deliverables, template architecture, and KPI frame.

Phase 2. Tooling and template build

Translate the approved model into operational workbooks and standard outputs.

Phase 3. Pilot and refinement

Test the model on live cases and correct failure points.

Phase 4. Rollout

Adopt the model as the standard operating system for future cohorts.

Success criteria and KPI framework

Process quality

Consistency of structured outputs, defensibility of major decisions, and reduction in reviewer-driven variability.

Speed and efficiency

Turnaround time from startup selection to first structured recommendation and onward decisions.

Startup outcomes

Progression to support activation, investor-readiness, and investor facilitation.

Mentor efficiency

Mentor time spent on support rather than administration, plus assignment quality and rework rate.

Risks and assumptions

Risks

  • The project grows beyond initial expectations. Response: phase the work and keep approval gates explicit.
  • The process becomes too complex. Response: keep the design operator-led and tool-backed.
  • Standardisation creates unnecessary overhead. Response: preserve controlled flexibility and low mentor admin.

Assumptions

  • Existing draft files are treated as exploratory material, not as the approved model.
  • The flowchart remains the visual backbone and is updated in parallel with the written scope.
